**Estado general:** v0.1.4 está mayormente sólida en la capa de servicios y persistencia
(todos los smoke tests pasan; el dominio de ventas, compras, cuentas corrientes, precios,
MP QR, license y LAN están bien aislados y testeados). El asar empaquetado contiene migraciones
y branding correctos. El backend (Fastify cloud), license manager y updater están bien tipados
y con fallbacks. La integración ESC/POS en ticket de venta y cajón monedero está bien
implementada y con fallback a `window.print()`.

**Áreas de mayor riesgo estático:** (1) Layout MDI — los reservados verticales del
`InternalWindow` no contemplan el `QuickAccessToolbar` ni la altura real del MenuBar, lo
cual permite que el clamp permita arrastrar ventanas debajo de la chrome y que el cálculo de
posicionado quede inconsistente con el contenedor Desktop. (2) UX del PDV — el split de pagos
se muestra por defecto incluso para ventas mono-medio, contradice la expectativa del comerciante
(95% de las ventas son 1 medio). (3) MercadoPago QR — la `webhookUrl` que se muestra al usuario
es literalmente `https://api.stockflow.com.ar/api/mp/webhook/TU_TENANT_ID` (hardcode),
impidiendo configurar el webhook real en el panel MP sin conocimiento técnico.

**Áreas sin issues bloqueantes:** ABM Artículos / Clientes / Proveedores / Familias / Usuarios,
Caja (apertura/cierre/movimientos), Cuentas corrientes (clientes y proveedores), Contabilidad
y Libro IVA (cards y export Excel), Consultas (5 pantallas), Wizard de actualización masiva
de precios con historial y rollback, Command Palette y GlobalSearchBar, Importación Excel
con preview/validación, BackupService, LicenseManager con master y readonly, auto-update,
hardware printer / cashDrawer / scale (servicios).

### CRITICAL

#### BUG-001: Webhook URL de MercadoPago QR es literal `TU_TENANT_ID`
- **Severidad:** CRITICAL
- **Módulo:** MercadoPago QR — Configuración
- **Archivo:** `apps/desktop/src/pages/ConfiguracionMercadoPago.tsx:59-60`
- **Descripción:** El componente arma el `webhookUrl` que el comerciante debe
  pegar en el panel de MercadoPago con un tenantId hardcodeado:
  ```ts
  const tenantId = 'TU_TENANT_ID'
  const webhookUrl = `https://api.stockflow.com.ar/api/mp/webhook/${tenantId}`
  ```
  No hay ningún `useQuery`/contexto que provea el tenantId real desde el
  licenseManager o `api.license.getState()`. El comerciante no técnico no sabe
  que tiene que reemplazarlo, y MP rechazará el webhook por 404 o lo enviará a
  un tenant inexistente.
- **Resultado esperado:** mostrar la URL real con el tenantId del comercio
  ya activado.
- **Resultado actual:** texto literal `TU_TENANT_ID` en el card de Estado.
- **Propuesta de fix:** leer `tenantId` de `useLicenseStatus`/`api.license.getState()`
  (existe `tenantName` cacheado) o agregar campo `tenantId` al estado de licencia;
  fallback "Activá tu licencia primero" si no hay tenant.

#### BUG-002: `InternalWindow.TOP_RESERVED` no contempla `QuickAccessToolbar`
- **Severidad:** CRITICAL (regresión introducida por P-MDI-TOOLBAR; rompe UX MDI)
- **Módulo:** Layout MDI / WindowManager
- **Archivo:** `apps/desktop/src/components/InternalWindow.tsx:27`
- **Descripción:**
  ```ts
  const TOP_RESERVED = 88 // MenuBar (48) + StatusBar (40)
  ```
  El comentario está desactualizado (MenuBar es `h-9` = 36px, no 48) y, más
  grave, la chrome real es: `MenuBar h-9` (36) + `QuickAccessToolbar h-20` (80)
  + `StatusBar h-10` (40) + opcional banner readOnly. Total ≈ 156–188 px.
  Además, las ventanas son `absolute` dentro de `Desktop` (que está `relative`
  con `flex-1`), por lo que `top: position.y` es relativo a Desktop — pero el
  clamp usa `globalThis.window.innerHeight` (viewport). Hay una inconsistencia
  de sistemas de coordenadas:
  - `clampPosition` limita `y >= TOP_RESERVED` en coords de viewport.
  - El render aplica `top: win.position.y` en coords del Desktop padre.
  Esto se traduce en que con `position.y=88` la ventana aparece a 88px DENTRO
  del Desktop (visualmente ≈ 244–276px del viewport), nunca llegando a tocar el
  StatusBar. El usuario percibe "hay un margen muerto arriba" y "no llega al
  fondo".
- **Resultado esperado:** la ventana puede flotar sobre toda el área Desktop,
  los clamps usan coords del Desktop, y los reservados reflejan la chrome real.
- **Resultado actual:** offset y clamps inconsistentes; los `maxY/maxX` de drag
  usan viewport pero los reservados están desactualizados.
- **Propuesta de fix:** usar `useRef` sobre el contenedor Desktop y leer
  `getBoundingClientRect()` para clamps; eliminar la constante `TOP_RESERVED`
  y derivar la altura real del Desktop. Alternativamente, sumar 96 a
  `TOP_RESERVED` para reflejar QuickAccessToolbar (h-20+border) — fix mínimo
  para no resolver el bug de coords mixtas. Reservar 28px adicionales cuando
  `licenseStatus === 'readOnly'` (banner visible).

#### BUG-003: Submenú "Ayuda" — 3 entradas distintas abren la misma página `configuracion`
- **Severidad:** CRITICAL (UX engañosa; el usuario no puede llegar a la sección que cree)
- **Módulo:** Layout MDI / MenuBar
- **Archivo:** `apps/desktop/src/components/MenuBar.tsx:144-153`
- **Descripción:** Las opciones "Hardware (Impresora, Balanza, Cajón)", "Backup"
  y "LAN Multi-caja" tienen `pageKey: 'configuracion'` sin parámetro de
  sección. Las tres abren la misma ventana de Configuración sin ningún
  `params` ni hash a la tab correspondiente. El usuario hace click en
  "Backup" y aterriza en la pestaña por defecto de Configuración (probablemente
  Hardware), confundiéndose.
- **Resultado esperado:** cada entrada debería abrir la window con
  `params: { tab: 'hardware' | 'backup' | 'lan' }` y la página debería leer
  ese param para seleccionar el tab inicial.
- **Resultado actual:** las 3 entradas son visualmente distintas pero
  funcionalmente iguales.
- **Propuesta de fix:** ampliar `MenuItem` con `params?: Record<string, unknown>`,
  propagarlo a `wm.openWindow({ pageKey, params })` y en `Configuracion.tsx`
  leer `useWindowSelf().params.tab` para inicializar `defaultValue` del `<Tabs>`.

### HIGH

#### BUG-004: PDV — el split de pagos se muestra por defecto, no hay dropdown "Forma de pago"
- **Severidad:** HIGH (UX core del negocio; 95% de las ventas son mono-medio)
- **Módulo:** PDV / Ventas
- **Archivo:** `apps/desktop/src/pages/Ventas.tsx:737-739`, `apps/desktop/src/components/PaymentSplitInput.tsx`
- **Descripción:** El componente `PaymentSplitInput` renderiza **N filas** (una
  por cada medio de pago activo) con un input vacío en cada una. Para una
  venta normal en efectivo el cajero tiene que tipear el monto en la fila
  "Efectivo" o tocar el botón secundario "Todo en Efectivo". No existe un
  Select / radio inicial "Forma de pago" con default "Efectivo" que
  autocomplete el monto y oculte el split mixto detrás de un toggle.
- **Resultado esperado (UX comercio real):** Select "Forma de pago" con
  default `Efectivo` que autocomplete el monto total al medio elegido +
  toggle "Pago mixto" que expone el split actual.
- **Resultado actual:** el cajero ve siempre N filas vacías; cualquier
  desliz tipográfico hace que `split.remaining > 0` y bloquea el botón
  "Confirmar venta".
- **Propuesta de fix:** envolver `PaymentSplitInput` en un `Tabs` /
  `RadioGroup` con dos modos: "Único medio" (Select de medio, monto =
  total) y "Mixto" (componente actual). Default "Único medio = Efectivo".
  Atajos: `F3` para alternar modo, `Enter` para confirmar.

#### BUG-005: Cierre de Caja — siempre usa `window.print()` aunque haya impresora térmica configurada
- **Severidad:** HIGH (regresión vs. comportamiento esperado de ESC/POS en P-HW-CIERRE)
- **Módulo:** Caja / Print
- **Archivo:** `apps/desktop/src/pages/Caja.tsx:11,129,218`
- **Descripción:** `Caja.tsx` importa `usePrintCashClose` (que rutea a
  `PrintProvider.print()` → `window.print()` del iframe HTML). El handler
  ESC/POS sí existe en el backend (`PrinterService.printCashCloseReport` y
  `api.hardware.printer.printCashClose`) pero NO se llama desde Caja.
- **Resultado esperado:** si hay `printerConfig`, mandar bytes ESC/POS;
  si no, `window.print()` como fallback (mismo patrón que en Ventas.tsx).
- **Resultado actual:** siempre dialog de impresión del SO con HTML.
- **Propuesta de fix:** replicar el patrón de `Ventas.tsx:398-444` en el
  handler de cerrar caja. Análogo para `HistorialCajas.tsx` (printCashClose
  por línea).

#### BUG-006: Compras — mismo problema D7 (split forzado por default)
- **Severidad:** HIGH
- **Módulo:** Compras
- **Archivo:** `apps/desktop/src/pages/Compras.tsx:278`

### MEDIUM

#### BUG-007: F11 no está mapeado (gap en F-keys F1..F10 + F12)
- **Severidad:** MEDIUM (polish)
- **Módulo:** Atajos MDI
- **Archivo:** `apps/desktop/src/lib/useMdiShortcuts.ts:82-93`
- **Descripción:** El regex `/^F([1-9]|10)$/` mapea F1..F10. F12 = logout.
  F11 queda sin asignar. La spec P-MDI-LAYOUT no lo requiere explícitamente,
  pero rompe la simetría visual del MenuBar (donde F11 podría usarse para
  algo como "Ayuda" o "Cmd Palette" alternativo).
- **Propuesta de fix:** asignar F11 a `wm.cycleFocus(1)` (alternativa a
  Cmd+Tab), o documentar explícitamente que está reservado.

#### BUG-008: Migración 0004 inexistente — gap en la numeración de Drizzle
- **Severidad:** MEDIUM (no rompe, pero confunde a quien revise)
- **Módulo:** Schema / Migrations
- **Archivos:** `packages/db/migrations/local/` (existen 0000, 0001, 0002, 0003, 0005, 0006)
- **Descripción:** El journal `meta/_journal.json` salta `idx: 4`. Drizzle
  no se queja porque va por idx y tags. Pero un nuevo dev / auditor pierde
  tiempo buscando el archivo.
- **Propuesta de fix:** renombrar `0005`→`0004` y `0006`→`0005` ANTES de
  que algún cliente productivo aplique migraciones más nuevas; o agregar un
  comentario `0004_INTENTIONALLY_SKIPPED.md` explicando.

#### BUG-009: `confirmar()` con MP QR — race entre `linkOrderToSale` y refetch
- **Severidad:** MEDIUM
- **Módulo:** PDV / MP QR
- **Archivo:** `apps/desktop/src/pages/Ventas.tsx:478`
- **Descripción:** Tras `createSale.mutateAsync` para el path MP QR, se llama
  `api.mpQr.linkOrderToSale(orderId, result.sale.id).catch(() => {})`. El
  `.catch(() => {})` silencia errores. Si el link falla, el comerciante
  no se entera y la trazabilidad MP↔Sale queda rota.
- **Propuesta de fix:** loggear el error (toast.warning) o reintentar 1
  vez antes de tragárselo.

#### BUG-010: Articulos toolbar — "Imprimir" siempre `window.print()` (UX correcto, pero sin guard)
- **Severidad:** MEDIUM (no es bug crítico — listar artículos en térmica no aplica)
- **Módulo:** ABM Artículos
- **Archivo:** `apps/desktop/src/pages/Articulos.tsx:502-504`
- **Descripción:** Imprimir el listado de artículos siempre usa
  `window.print()`. Esto es **correcto** porque una lista no entra en
  papel térmico, pero conviene documentarlo en el tooltip ("Imprimir
  vista actual en A4"). Si en el futuro se añade "imprimir etiqueta de
  góndola por artículo" sí se requiere ESC/POS.
- **Propuesta de fix:** dejar como está pero clarificar el tooltip y, si
  se quiere etiquetas térmicas, agregar un botón distinto.
